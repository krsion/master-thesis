# Conclusion {#chap:conclusion}

This thesis investigated approaches to collaborative editing in the Denicek system --- a computational substrate for document-oriented end-user programming. We evaluated two CRDT libraries (Automerge, Loro) and ultimately built a custom **pure op-based CRDT** whose state is a grow-only event graph and whose document is a pure view function over that graph.

Automerge's flat map representation lacked atomic move operations, making the wrap edit --- one of Denicek's core structural operations --- unreliable under concurrent editing. Loro solved this with a native movable tree CRDT, but its opaque ID-based addressing proved incompatible with Denicek's path-based programming model: recorded edits could not use relative references, and replayed edits could not be retargeted through concurrent structural changes.

The final implementation, framed as a pure operation-based CRDT [@baquero2017pureop] with an event-graph architecture inspired by Eg-walker [@gentle2025egwalker], stores every edit as an immutable event in a grow-only DAG and materializes documents by replaying events in a deterministic topological order with selector-rewriting transformations for concurrent structural edits. Convergence follows directly from the Baquero framework: the event set is a G-Set, the view function is pure, and the proof ([@Sec:crdt-framing]) reduces to showing that `materialize` is deterministic. The hard part --- and the main engineering contribution --- is **intention preservation**: ensuring that references stay valid through structural edits, that wildcard operations expand over concurrent inserts, and that recorded edit sequences can be replayed after schema evolution.

The implementation was validated on five formative examples demonstrating custom primitive edits, formula recomputation, the composer pattern, structural transformation (list to table), and concurrent editing with fork-and-merge. All examples pass their tests and are available as interactive demos in the deployed web application.

## Future work {#sec:future-work}

**Optimized materialization.** The current implementation caches the materialized document for linear extensions and saves checkpoints at fork points so that merges resume from the fork rather than replaying from scratch. However, the `resolveAgainst` step still scans all prior events for each replayed event, giving $O(N^2)$ in the worst case. A finer-grained optimization --- for instance, indexing prior events by their structural-edit target prefix so that non-overlapping priors can be skipped --- would reduce the per-event resolution cost. The current implementation could serve as a *reference oracle* --- a correct implementation used to validate the behavior of faster versions: generate random edit sequences, run both implementations, and compare outputs. This approach could be further strengthened by model-guided fuzzing [@ozkan2025modelfuzz], which uses an abstract formal model (such as the reference implementation) to guide test generation and achieve higher coverage of subtle concurrency-related bugs than random fuzzing alone.

**Extended mechanical verification.** A TLA+ specification covering five edit types (Add, Rename, Insert, Delete, WrapRecord) on a flat record model has been verified by TLC model checking, confirming strong eventual consistency for all reachable states of the bounded model. Extending the specification to cover nested trees, list-based operations, and wildcard selectors would provide stronger guarantees but would require careful state-space management --- adding a single edit type roughly triples the state space. A mechanized proof in Isabelle/HOL or Coq along the lines of the pure op-based framework of Baquero et al. [@baquero2017pureop] is a further, more ambitious step that would cover the unbounded case.

**Character-level text editing.** Integrating a text CRDT such as Fugue for primitive string values would enable character-level collaborative editing, replacing the current atomic last-writer-wins semantics for strings.

**mydatnicek.** A data-oriented variant of Denicek, built on the optimized core, could handle larger datasets (spreadsheets, databases) while preserving local-first collaborative editing. The tagged tree model naturally extends to tabular data with formula columns.

**Event compaction.** The server-side compaction mechanism ([@Sec:compaction-offline]) materializes the document and discards old events once all active peers have acknowledged a common frontier. However, compaction currently uses a simple activity-timeout heuristic to determine which peers are "active" --- peers inactive for more than five minutes are excluded from the consensus check. A more robust distributed acknowledgment protocol would allow the server to compact more aggressively while guaranteeing that no peer loses unsynced edits. Additionally, compacted offline edits that conflict with the new state are currently dropped silently; surfacing these as user-visible conflicts would improve the user experience.
