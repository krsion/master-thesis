
# Introduction {-}

*Document-oriented programming systems* [@petricek2025designchoices] are programming environments built around the metaphor of a document editor. The user interacts with a structured document that contains both code and data; they modify the document, trigger computations, and view results --- all from a unified interface. Spreadsheets are the most familiar example, but the paradigm also includes data-science notebooks and recent research systems. What distinguishes document-oriented programming from conventional programming is that the program *is* the document: there is no separate source file, build step, or hidden execution state.

Denicek [@petricek2025denicek] is one such system --- a computational substrate for document-oriented end-user programming. Its documents are tagged trees (records, lists, primitives, and references) addressed by path-based selectors such as `speakers/0/name` or `items/*`. Users program by recording edit sequences and replaying them (programming by demonstration). The original Denicek uses Operational Transformation (OT) for collaboration, but OT is notoriously fragile: several published algorithms were later proven incorrect [@imine2003proving], and the number of pairwise rules grows with each new edit type.

When a document-oriented programming system supports collaborative editing, it faces a distributed systems challenge: multiple peers modify the same document concurrently, and the system must merge their edits so that all peers converge to the same state. *Local-first software* [@kleppmann2019localfirst] makes this explicit --- each device holds a full copy, edits apply immediately, and synchronization happens in the background. Two main families of algorithms address convergence: Operational Transformation (OT), which transforms operations pairwise, and *Conflict-free Replicated Data Types* (CRDTs) [@shapiro2011crdt], which structure the data so that concurrent modifications can be merged without conflicts. CRDTs guarantee *strong eventual consistency*: any two replicas that have received the same set of updates will be in the same state.

This thesis investigates more robust collaborative editing for Denicek. We evaluate two CRDT libraries --- Automerge and Loro --- and build a custom design, describing the trade-offs of each. The result, *mydenicek*, is a **pure operation-based CRDT** [@baquero2017pureop]: edits are stored in a grow-only event graph, and the document is computed by a deterministic *eval* function that transforms concurrent edits during topological replay.

Convergence --- all peers reaching the same state --- is the baseline guarantee. The harder problem is **intention preservation** [@sun1998achieving]: the effect of an operation in the merged document should match the effect that the operation had on the document state from which it was generated. Convergence requires that replicas eventually see the same event set and evaluate it deterministically; intention preservation is what makes the merged result *meaningful*. For tree-structured documents with path-based selectors, this means that when the document structure changes concurrently --- fields are renamed, nodes are wrapped, items are inserted --- each operation must still reach its intended target and produce its intended effect. Defining a clear model of user intent for each class of concurrent interaction, and designing merge rules that preserve it, is the central challenge of this thesis.

The main contributions are:

- Evaluation of document-based representations in two CRDT libraries (Automerge, Loro) for tree-structured collaborative editing, identifying concrete limitations: the concurrent wrap problem and the retargeting problem ([@Chap:background]).
- A novel pure operation-based CRDT for tagged-tree documents with path-based selectors, wildcards, relative references, and strict indices, together with a proof of strong eventual consistency ([@Chap:implementation]).
- Semantics that preserves user intent across concurrent edits, including explicit invariants: wildcard edits affect concurrent insertions, references survive structural changes, and indices shift correctly ([@Chap:evaluation]).
- Programming-by-demonstration support built on the same underlying event graph: recorded edits are retargeted through later structural changes, enabling replay after schema evolution ([@Chap:evaluation]).

## Thesis structure {-}

[@Chap:background] covers CRDTs, OT, causality, and related work. [@Chap:implementation] presents the mydenicek CRDT design and implementation. [@Chap:evaluation] evaluates correctness and performance on formative examples. [@Chap:conclusion] concludes with future work.
